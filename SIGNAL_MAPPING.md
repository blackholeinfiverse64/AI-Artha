# SIGNAL_MAPPING.md — Artha Signal Contract Mapping

## Signal ID Format

```
SIG_{SOURCE}_{EVENT}
```

Examples: `SIG_GST_MISMATCH`, `SIG_TDS_MISSING`, `SIG_LEDGER_IMBALANCE`

## Trace ID Format

```
TRC-{YYYYMMDD}-{UUID_SHORT}
```

Generated by `period.util.js → buildTraceId()` (currently plain UUID).
Proposed format enforced in `signalEngine.service.js`.

## Severity Enum

| Value | Meaning |
|-------|---------|
| `CRITICAL` | Blocks filing / legal risk |
| `HIGH` | Requires immediate action |
| `MEDIUM` | Requires action within period |
| `LOW` | Informational / monitor |

## Signal Source Map

### Source: GST_ENGINE

| Event | signal_id | Trigger | Severity |
|-------|-----------|---------|----------|
| GST rate not in [0,5,12,18,28] | `SIG_GST_INVALID_RATE` | `gstEngine.calculateGSTBreakdown()` throws | HIGH |
| IGST mixed with CGST/SGST | `SIG_GST_MIXED_TAX_TYPE` | `ledger.service.validateGSTDetails()` | HIGH |
| Invoice tax amount ≠ GST calculation | `SIG_GST_MISMATCH` | `invoice.service.sendInvoice()` tolerance check | HIGH |
| Expense tax amount ≠ GST calculation | `SIG_GST_EXPENSE_MISMATCH` | `expense.service.recordExpense()` tolerance check | HIGH |
| Company state missing | `SIG_GST_COMPANY_STATE_MISSING` | `invoice.service.sendInvoice()` | CRITICAL |
| Customer state missing | `SIG_GST_CUSTOMER_STATE_MISSING` | `invoice.service.sendInvoice()` | HIGH |
| Missing GSTIN for B2B supply | `SIG_GST_MISSING_GSTIN` | `compliance/validation.service.validateGSTR1()` | HIGH |
| Invoice date outside filing period | `SIG_GST_PERIOD_MISMATCH` | `compliance/validation.service.validateGSTR1()` | MEDIUM |
| Duplicate invoice reference in filing | `SIG_GST_DUPLICATE_REFERENCE` | `compliance/validation.service.validateGSTR1()` | HIGH |
| Negative net tax liability | `SIG_GST_NEGATIVE_LIABILITY` | `compliance/validation.service.validateGSTR3B()` | HIGH |

### Source: TDS_ENGINE

| Event | signal_id | Trigger | Severity |
|-------|-----------|---------|----------|
| TDS amount > payment amount | `SIG_TDS_EXCESS_DEDUCTION` | `ledger.service.validateTDS()` | HIGH |
| Missing deductee PAN | `SIG_TDS_MISSING_PAN` | `compliance/validation.service.validateForm26Q()` | HIGH |
| Invalid TDS section | `SIG_TDS_INVALID_SECTION` | `compliance/validation.service.validateForm26Q()` | HIGH |
| Missing challan linkage | `SIG_TDS_MISSING_CHALLAN` | `compliance/validation.service.validateForm26Q()` | HIGH |
| Missing employee PAN (Form 24Q) | `SIG_TDS_MISSING_EMPLOYEE_PAN` | `compliance/validation.service.validateForm24Q()` | HIGH |

### Source: LEDGER

| Event | signal_id | Trigger | Severity |
|-------|-----------|---------|----------|
| Debits ≠ Credits | `SIG_LEDGER_IMBALANCE` | `ledger.service.validateJournal()` | CRITICAL |
| Hash mismatch on entry | `SIG_LEDGER_HASH_TAMPER` | `ledger.service.verifyLedgerChain()` | CRITICAL |
| Chain linkage broken | `SIG_LEDGER_CHAIN_BREAK` | `ledger.service.verifyLedgerChain()` | CRITICAL |
| Account inactive or not found | `SIG_LEDGER_INVALID_ACCOUNT` | `ledger.service.validateAccounts()` | HIGH |
| Line has both debit and credit | `SIG_LEDGER_LINE_INTEGRITY` | `ledger.service.validateLineIntegrity()` | HIGH |
| Net cash flow negative | `SIG_CASHFLOW_NEGATIVE` | `signalEngine.getSignalSnapshot()` | HIGH |

### Source: INVOICE

| Event | signal_id | Trigger | Severity |
|-------|-----------|---------|----------|
| Invoice overdue (past dueDate, unpaid) | `SIG_INVOICE_OVERDUE` | Aged receivables / status = overdue | MEDIUM |
| Payment exceeds amount due | `SIG_INVOICE_OVERPAYMENT` | `invoice.service.recordPayment()` | HIGH |
| Invoice cancelled with payments | `SIG_INVOICE_CANCEL_WITH_PAYMENT` | `invoice.service.cancelInvoice()` | HIGH |

### Source: EXPENSE

| Event | signal_id | Trigger | Severity |
|-------|-----------|---------|----------|
| Expense recorded without approval | `SIG_EXPENSE_SKIP_APPROVAL` | `expense.service.recordExpense()` status check | HIGH |
| Auto-record after approval failed | `SIG_EXPENSE_RECORD_FAILED` | `expense.service.approveExpense()` catch block | MEDIUM |

### Source: COMPLIANCE_FILING

| Event | signal_id | Trigger | Severity |
|-------|-----------|---------|----------|
| Filing not ready (has errors) | `SIG_FILING_NOT_READY` | `validation.service` → `filing_ready: false` | HIGH |
| Filing generated (informational) | `SIG_FILING_GENERATED` | `gstStatutory` / `tdsStatutory` service | LOW |

## Signal → Compliance → Ledger Traceability Chain

```
SIG_GST_MISMATCH
  │  context.invoice_id → Invoice._id
  │  context.trace_id   → JournalEntry.trace_id
  ▼
ComplianceValidationLog
  │  filingId → ComplianceFiling.filingId
  │  traceId  → ComplianceFiling.traceId
  ▼
ComplianceFiling
  │  sourceTransactions[].sourceId → JournalEntry._id
  ▼
JournalEntry (POSTED)
  │  lines[].account → ChartOfAccounts
  │  prevHash / hash → chain integrity
  │  gstDetails[]    → GST breakdown
  ▼
LedgerEntry (per line)
  │  journal_id → JournalEntry._id
  │  account_id → ChartOfAccounts._id
  ▼
AccountBalance (running total)
```

Every signal must carry enough context to reconstruct this chain.

## Signal Persistence

Signals are written to `ComplianceSignal` model:

```js
{
  signal_id:      "SIG-<uuid>",          // auto-generated by model
  trace_id:       "TRC-YYYYMMDD-<uuid>", // from originating transaction
  source:         "ARTHA",               // immutable
  type:           "SIG_GST_MISMATCH",    // from enum above
  severity:       "HIGH",
  context:        { ... },               // entity-specific
  recommendation: "...",
  created_at:     ISODate
}
```

Currently: `ComplianceSignal` model exists but **no service writes to it**.
Required: `signalEngine.service.js` must call `ComplianceSignal.create()` for each detected signal.
